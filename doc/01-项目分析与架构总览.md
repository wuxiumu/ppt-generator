## 01 - 项目分析与架构总览

### 需求拆解

这道题表面是"生成PPT"，实际考核三件事的交集：

**叙事连贯性（"一套"而非"一堆"）**：25-30张幻灯片必须有起承转合的叙事弧线——开头引入、中段展开、收尾升华，而不是30张互不相关的卡片。这是最核心也最容易被忽视的难点。

**视觉一致性**：整套PPT像同一个设计师做的，配色、排版、字号、留白、图表风格全部统一。

**成本与速度的工程约束**：单份 < $10（美观极致版）或更低（平衡版），< 30分钟生成。

### 三个考核维度的优先级判断

美观度是天花板——做到什么程度直接决定分数上限。成本和时间是约束条件——必须在约束内最大化美观度。叙事连贯性是隐藏加分项——"一套"和"一堆"的差距，评委一眼就能看出来。

我的优先级排序：**叙事连贯 > 视觉一致 > 美观度上限 > 成本优化 > 速度优化**。

### 核心架构：三阶段流水线

```
JSON输入
  │
  ▼
┌─────────────────────────────────┐
│  Stage 1: 叙事规划（Planner）     │  ← 强模型（Claude Sonnet / GPT-4o）
│  输入: topic + brief + audience  │
│  输出: 完整叙事大纲 + 风格定义    │
│  包含: 三幕结构、每幕主题、       │
│        每张slide的叙事角色、       │
│        配色方案、排版风格          │
└──────────┬──────────────────────┘
           │
           ▼
┌─────────────────────────────────┐
│  Stage 2: 内容生成（Writer）      │  ← 中等模型（GPT-4o-mini / Claude Haiku）
│  逐张生成slide内容                │
│  每张slide知道自己在叙事中的角色   │
│  输出: 结构化JSON（标题/正文/     │
│        图表数据/图片描述/讲稿）    │
└──────────┬──────────────────────┘
           │
           ▼
┌─────────────────────────────────┐
│  Stage 3: 渲染引擎（Renderer）    │  ← 纯代码，无LLM调用
│  结构化数据 → python-pptx         │
│  应用设计系统：布局/配色/字体/     │
│  间距/图表/图片/动画             │
└──────────┬──────────────────────┘
           │
           ▼
      .pptx 文件
```

### 为什么是三阶段而非一阶段

**一阶段（一次LLM调用生成所有内容）**的问题：上下文窗口限制（25-30张完整slide的内容量可能超过输出token上限）、无法精细控制叙事节奏、无法做风格一致性校验。

**三阶段的优势**：Stage 1 只做规划不做内容，token开销小但决定了全局质量；Stage 2 每张slide可以并行生成，带上下文但任务明确；Stage 3 纯代码渲染，100%确定性，不烧token。

### 数据流详解

```
输入 JSON
    │
    ├─→ [Stage 1] Planner LLM ─→ 全局规划文档（~3000 tokens）
    │         │
    │         ├─ narrative_arc: { act1: "引入", act2: "展开", act3: "收尾" }
    │         ├─ slide_plan: [ { slide_num, narrative_role, layout_type,
    │         │                  content_brief, visual_notes } ]
    │         ├─ style_spec: { palette, fonts, tone, visual_metaphor }
    │         └─ image_prompts: [ { slide_num, description, style } ]
    │
    ├─→ [Stage 2] Writer LLM × N（并行）
    │         │  每张slide一个调用，prompt包含：
    │         │  - 该slide的叙事角色和content_brief
    │         │  - 前一张和后一张的摘要（保证过渡自然）
    │         │  - style_spec（保证语气统一）
    │         │
    │         └─→ slide_content: { title, subtitle, bullets, 
    │              body_text, chart_data, speaker_notes }
    │
    ├─→ [Stage 2.5] 图片生成（可选，美观极致版）
    │         │  并行调用 DALL-E 3 / Flux / Ideogram
    │         └─→ images: [ { slide_num, file_path } ]
    │
    └─→ [Stage 3] Renderer（纯Python代码）
              │  - 根据 layout_type 选择模板
              │  - 根据 style_spec 应用配色
              │  - 插入图表、图片、文本
              │  - 添加页码、章节分隔页
              └─→ final.pptx
```

### 技术栈选择

**语言：Python 3.11+**
- python-pptx 是最成熟的PPTX生成库，API稳定，社区大
- Pillow 处理图片
- matplotlib/plotly 生成图表再导出为图片嵌入

**LLM API：OpenAI + Anthropic 双供应商**
- 避免单一供应商依赖
- 可以根据成本动态选择

**图片生成（美观极致版）：DALL-E 3 或 Flux**
- DALL-E 3 质量稳定但稍贵（$0.04/张）
- Flux 开源可本地部署，成本趋近于零

### 关键技术风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 叙事断裂：相邻slide衔接生硬 | 核心扣分点 | Stage 2 prompt 注入前后slide摘要；后处理校验过渡页 |
| 视觉不一致：字号/间距漂移 | 看起来像拼凑的 | 设计系统硬编码，LLM不控制视觉参数 |
| 内容溢出：某张slide文字太多 | 排版崩坏 | 渲染前做字数校验+自动分页 |
| 图片风格漂移 | 不同slide配图风格差异大 | 统一image style prompt前缀 |
| API超时/限流 | 生成时间超标 | 并行请求 + 超时fallback + 缓存 |
