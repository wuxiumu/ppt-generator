**为什么是三阶段而非一阶段？**

一阶段（一次 LLM 调用生成所有内容）的问题：上下文窗口限制（25-30 张完整 slide 可能超过输出 token 上限）、无法精细控制叙事节奏、无法做风格一致性校验。

三阶段的优势：Stage 1 只做规划不做内容，token 开销小但决定了全局质量上限；Stage 2 每张 slide 可并行生成，带上下文但任务明确；Stage 3 纯代码渲染，100% 确定性，不烧 token。

## 模型选型

### 对比矩阵

| 模型 | 输入 $/M | 输出 $/M | 规划质量 | 内容质量 | 速度 | 总成本/套 |
|------|---------|---------|---------|---------|------|----------|
| DeepSeek Chat | $0.27 | $1.10 | ★★★★ | ★★★★ | **~30s** | **$0.012** |
| Qwen 3.7 Plus | $0.80 | $4.00 | ★★★★ | ★★★★ | ~336s | $0.073 |
| GPT-4o | $2.50 | $10.00 | ★★★★★ | ★★★★★ | ~45s | $0.15 |
| Claude 3.5 Sonnet | $3.00 | $15.00 | ★★★★★ | ★★★★★ | ~40s | $0.20 |

### 选型理由

**默认 DeepSeek**：在成本约束（<$10/套）内速度最快（30s），质量足够好。5 套总成本仅 $0.059，远低于预算。叙事规划能力良好，5 个 demo 的调色板选择和叙事类型匹配全部正确。

**千问实测**：通过 DashScope 的 Anthropic 兼容端点接入。内容质量与 DeepSeek 相当，但速度慢 10 倍（~336s），且输出 token 计算方式不同（62K vs 7.7K），成本更高。适合对中文质量有更高要求的场景。

**GPT-4o / Claude**：规划质量最高，适合美观极致版。成本仍然远低于 $10 限制。

### 模型分级策略

最终没有采用原设计的"Planner 用强模型、Writer 用弱模型"的分级策略，原因是：
- DeepSeek 本身就足够便宜，分级节省的成本微乎其微（<$0.005）
- 统一模型避免了风格不一致的风险
- 代码更简单

## 风格一致性怎么保的

这是核心难点，我的解法是 **"LLM 不碰视觉"**：

### 1. 渲染引擎硬编码所有样式

```python
# renderer.py — 所有视觉参数由代码控制，LLM 输出中没有任何样式信息
LAYOUTS = {
    "bullets": {
        "title": {"p": (0.08, 0.06), "s": (0.82, 0.12)},
        "body":  {"p": (0.12, 0.22), "s": (0.78, 0.66)},
    }, ...
}
```

LLM 的输出只有结构化数据（标题、要点、正文），不包含任何颜色、字号、位置信息。渲染引擎根据 layout type 查找预定义的布局规格，确保每张 slide 的视觉参数 100% 一致。

### 2. 调色板集中管理

5 套调色板定义在 `PALETTES` 字典中，每套包含 11 个语义化的颜色 token（bg, text, text2, accent1-4, surface, divider 等）。Planner 只选择调色板名称（如 "tech"），不生成任何颜色值。

### 3. 布局节奏自动控制

`fix_layout_rhythm()` 函数在 Planner 输出后自动扫描并修复连续相同布局：
- 检测到 3+ 张相同 layout → 将第 3 张改为不同于前后的替代布局
- 替代布局从 `["statement", "big_number", "bullets", "comparison"]` 中选择
- 5 套 demo 中触发了 6 次修复，平均每套 1.2 次

### 4. HTML 和 PPTX 共享同一套配色

两个渲染引擎使用相同的 `PALETTES` 字典，确保同一主题的 PPTX 和 HTML 版本颜色一致。

## 多样性怎么保的

不同输入的产出不能像同一个模板填空。解法：

### 1. 5 种叙事模板自动匹配

Planner 根据 topic + brief + audience 自动选择叙事类型：

| 主题 | 匹配的叙事类型 | 差异体现 |
|------|-------------|---------|
| Python 入门 | tutorial（渐进式） | 概念→示例→练习→综合 |
| 年度复盘 | timeline（时间线） | Q1→Q2→Q3→Q4→展望 |
| 咖啡豆 | guide（决策树式） | 需求→维度→对比→推荐 |
| Rust 重写 | pitch（问题-方案） | 痛点→方案→证据→行动 |
| 京都周末 | itinerary（行程式） | Day1上午→下午→Day2→Tips |

每种叙事类型的三幕比例、子章节划分、布局偏好都不同。

### 2. 调色板根据主题自动选择

5 个 demo 分别命中了 4 种不同调色板（tech, personal, warm×2, corporate），视觉风格完全不同。

### 3. Writer prompt 注入上下文

每张 slide 的 Writer prompt 包含前一张和后一张的 content_brief，确保内容过渡自然，不会出现"每张 slide 都是独立段落"的问题。

### 4. tone 参数影响语气

Planner 生成的 tone（如 "友好鼓励" / "专业严谨" / "温暖真诚"）会传递给 Writer，影响所有 slide 的措辞风格。

## 踩坑和取舍

### 试过被否的方案

**方案 1：一次 LLM 调用生成所有 slide 内容**
- 问题：25-30 张 slide 的内容量超过 DeepSeek 的 4096 output token 限制
- 改为：Planner 只输出规划（~3000 tokens），Writer 逐张并行生成

**方案 2：python-pptx 原生图表**
- 问题：原生图表样式难以统一控制，不同 Office 版本渲染不一致
- 改为：matplotlib 生成图表图片后嵌入（最终版未实现，留作后续迭代）

**方案 3：DALL-E 生成配图**
- 问题：每张图 $0.04-0.08，10 张图 $0.4-0.8，成本可接受但增加 2-3 分钟等待
- 决定：MVP 阶段不用 AI 生图，用纯色+几何装饰代替。美观极致版再加入
- 取舍：视觉丰富度 vs 生成速度

**方案 4：16:9 固定比例的 HTML**
- 问题：在大屏显示器上不全屏，两侧留黑边
- 改为：`100vw × 100vh` 全屏 + 内容区 `max-width: 1200px` 居中

**方案 5：Next.js / React 前端**
- 问题：构建工具链复杂，部署门槛高，对小白不友好
- 改为：纯 HTML/CSS/JS 单文件 SPA，Flask 直接 serve，零构建

### 关键技术决策

**布局节奏修复放在 Planner 后而非 Writer 后**：如果在 Writer 后修复，已经生成的内容需要丢弃重写。放在 Planner 后，只改规划不改内容，Writer 按修正后的布局生成。

**Writer 并发数限制为 5**：DeepSeek API 有速率限制（60 RPM for tier 2），28 张 slide 以 5 并发需要 ~6 轮，每轮 ~2s，总共 ~12s。

**Anthropic SDK 支持千问**：千问的 DashScope 提供了 Anthropic 兼容端点，但实际调用速度慢（单次 ~10s），且 token 计算方式与 OpenAI 不同。通过 `LLMClient` 的 provider 参数统一封装。

## AI 协作复盘

### AI 提出且我采纳的决策

1. **三阶段流水线架构**：AI 建议将生成拆分为 Planner → Writer → Renderer 三阶段，而非一次性生成。这个建议非常关键，解决了 token 限制和叙事控制的问题。

2. **布局节奏修复器**：AI 建议添加 `fix_layout_rhythm()` 后处理函数来自动修复连续相同布局。实测 5 套 demo 中触发了 6 次，证明这个机制是必要的。

3. **HTML 作为第二输出格式**：AI 建议除了 PPTX 外增加 HTML 输出，用 CSS scroll-snap 实现翻页。HTML 的视觉效果远超 python-pptx 的能力上限。

### AI 提出但我推翻的决策

1. **分级模型策略**：AI 建议 Planner 用 Claude Sonnet、Writer 用 GPT-4o-mini。我推翻了，因为 DeepSeek 本身就足够便宜（$0.012/套），分级节省的成本微乎其微，反而增加了代码复杂度和风格不一致的风险。

2. **复杂的验证系统**：AI 最初设计了一个 200+ 行的 `NarrativeValidator`，检查主题衔接、内容密度、过渡类型等。我精简为只检查两个核心指标（连续布局、标题非空），因为过度校验会误杀好内容。

3. **外部配置文件**：AI 建议将调色板和布局规格放在 YAML 配置文件中。我推翻了，直接在 Python 代码中定义更简单，减少了一个依赖（pyyaml）和文件加载的复杂度。

### AI 跑偏我拽回来的场景

**场景 1：PPTX 渲染的三重引号 bug**

AI 在生成 statement 布局代码时写了 `r0.text = """`（一个左引号字符），Python 将其解释为三重引号字符串的开始，导致整个文件解析失败。我发现后改为 `r0.text = "\u201c"`（Unicode 转义）。

教训：AI 生成代码时要特别注意字符串字面量中的特殊字符。

**场景 2：HTML 不全屏问题**

AI 最初用 `min(100vw, 177.78vh)` 维持 16:9 比例。用户反馈在大屏上两侧有黑边。我改为 `100vw × 100vh` + 内容区居中，解决了问题。

教训：AI 倾向于"技术上正确"（维持比例），但忽略了用户的实际使用场景（全屏演示）。

**场景 3：千问 Anthropic 端点超时**

AI 建议直接使用千问的 Anthropic 兼容端点，但没有考虑超时问题。首次测试时 2 分钟超时，后来发现单次调用需要 ~10s，全量生成需要 5-6 分钟。我增加了 10 分钟的超时设置。

教训：对不熟悉的服务端点要先做单次调用测试，再上全量。

### 我的核验/兜底措施

1. **每次 LLM 输出都做 JSON 解析容错**：`parse_json()` 函数处理 markdown 代码块包裹的情况
2. **Writer 单次失败不阻断全流程**：JSON 解析失败的 slide 使用备选内容（brief 截断为标题）
3. **规划不足 25 张时自动补充**：Planner 输出不足 25 张时，自动发起第二轮 LLM 调用补充
4. **布局节奏后处理**：自动修复连续相同布局，不依赖 LLM 的规划完美度
5. **成本追踪**：`CostTracker` 记录每次 LLM 调用的 token 消耗，实时显示总成本
6. **Debug 输出**：每次生成的 plan.json 和 contents.json 保存到 debug/ 目录，方便排查问题
